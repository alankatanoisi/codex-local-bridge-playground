3 · Judgment: which plan does the code actually support?

(a) Shipped roadmap — boundary translation PREMISE BROKEN

Its core claim — translate only in model-client.js, everything else untouched — assumed the dialect lived in ~4 files. It lives in 19. Worse, per the protocol notes, Codex reasoning items are opaque and must round-trip losslessly; squeezing them through Anthropic thinking/signature blocks and back is a permanent lossy adapter. Translation would also have to fabricate tool_use_id ↔ call_id mappings that then leak into transcripts, ledger repair, and replay. Cheapest to start, most expensive to live with.

(b) Fable — native-first items rewrite RIGHT DIRECTION, UNDER-SCOPED

Matches the code's reality: codex-transport.js already thinks in Responses items, so making the internal state native removes the translation layer instead of adding one. But its staged file list (model-client / run.js / tool-pipeline / context-compactor) misses roughly half the actual footprint: recovery (run-manifest, ledger-repair, replay-simulator), golden-eval, archive/*, human-log, kernel/contract.js, and the persisted transcript/session schema. Unbudgeted, those become mid-rewrite surprises.

(c) Codex — ModelRuntime + new :11438 bridge OVER-ENGINEERED NOW

A neutral runtime abstraction for a single provider is speculative generality, and a second local bridge duplicates what codex-transport.js already does in-process. The critique doc's call to discard this for now is confirmed by the code.

Gaps no plan addresses

  • No function_call SSE fixture. The transport's tool-call parsing — the heart of an agent runner — has zero test coverage. Only a pong (text) stream is fixtured.
  • Transcript/session schema versioning. No plan decides whether old Anthropic-shaped sessions are migrated, converted on read, or cleanly abandoned; resume/replay/ledger-repair all depend on the answer.
  • Golden-eval assertions encode Anthropic shapes; they'll need a native-items rewrite or they'll pass/fail meaninglessly during migration.
  • No end-to-end spike. Auth (CODEX_ACCESS_TOKEN), streaming-only quirks, and rate-limit behavior have never been exercised through the real loop — only through the smoke script.

4 · Top 5 pre-build actions (in order)

  1. Capture real fixtures: record live function_call, reasoning-item, multi-tool, and error/429 SSE streams into test/runner/fixtures/codex/; extend codex-transport.test.js against them. Nothing else is trustworthy until this exists.
  2. Write items.js + a one-page internal item-schema contract first (Fable's step 1), including an explicit transcript/session schemaVersion: 2 decision: migrate, convert-on-read, or clean break. Get this reviewed before touching run.js.
  3. Re-scope the Fable plan to the verified 19-file inventory: add recovery/replay/ledger-repair, golden-eval, archive, human-log, and kernel/contract as named stages (or explicitly quarantine them behind a converter with a deletion date).
  4. Thin vertical spike: wire codex-transport.js into model-client.js behind a flag and run one real read-only agent loop end-to-end (auth, stream, single tool call) before any rewrite of run.js. De-risks the unknowns cheapest.
  5. Hygiene: delete verify-done, verify-*.log, FOLDER-STRUCTURE.md; declare docs/lab-notes/codex-protocol-notes.md the pinned wire contract; note that full npm test runtime (>30s) needs a CI-friendly split.
