How to read this roadmap

Both assessments reached the same core verdict: the builder's top half is genuinely beginner-friendly (presets, the permission-style dropdown, the chaos gate), but it has real shipping bugs, its hand-copied data has drifted out of sync with the runner, and its Advanced panel is an expert-sized wall of undefined jargon. The plan below fixes correctness first, honesty second, and beginner experience third — so that every later UX improvement is built on a page that tells the truth.

Each item carries pills: Bug / Drift / UX for the kind of problem, Quick / Medium / Bigger for rough effort, and a status pill we will flip as work lands. "A" and "B" mark which assessment(s) proposed it (A = command-builder-assessment.html, B = command-builder-assessment-claude.html).

Phase 1 — Correctness bugs (do these first)

Small, low-risk fixes to defects that ship today. Each is independently testable by reloading the page and comparing the generated command. Nothing here touches the runner or bridge.

1.1 Restore the missing .layout CSS selector BugQuick DoneA + B

Around lines 68–73 of the stylesheet, the .layout { line was lost, leaving orphaned grid declarations (display: grid; grid-template-columns: 1fr 1fr; …) that the browser silently discards. The form and the sticky command panel were designed to sit side by side; today the page stacks in one column and the live-preview value of the builder is lost. One-line fix; the @media (max-width: 700px) block below it already expects the selector to exist.

Done when: desktop view shows form left, sticky command panel right; narrow view still stacks.

1.2 Fix the "Verify after edits" preset so its flags actually apply BugQuick DoneA

The verify preset declares allowShell: true, testWatch: true, but applyPreset() ignores those keys whenever a preset also sets a permissionStyle. The preset's own prompt promises --test-watch results, yet the generated command contains neither flag (and --test-watch requires --allow-shell at runtime anyway). Either add allowShell/testWatch to the post-style whitelist, or switch the preset to the edit-shell style — and check the updateUiState() ordering so shell-dependent boxes don't immediately un-check testWatch.

Done when: clicking the preset yields a command containing both --allow-shell and --test-watch.

1.3 De-duplicate --session-id / --session-path BugQuick DoneA

buildCommandParts() pushes the session flags once near the top and again inside the replayMode/repairMode branches, so replay/repair commands contain the flag twice. It happens to work (Node's parseArgs takes the last value), but a duplicated flag in a teaching tool erodes a beginner's trust. Remove the duplicate pushes in the replay/repair branches.

Done when: a replay command with a session id contains each session flag exactly once.

1.4 Harden shellQuote() against shell expansion BugQuick DoneA

When a prompt contains a single quote, the builder switches to double-quote wrapping without escaping $, backticks, or \ — so a prompt like don't touch $HOME gets $HOME expanded by the shell before the runner ever sees it. Use proper POSIX single-quote escaping (the '…'"'"'…' pattern) so pasted free text can never be interpreted by the shell. Beginners paste free text; the quoting must be bulletproof.

Done when: prompts containing ', $, and backticks round-trip to the runner byte-for-byte.

Phase 2 — Make the builder tell the truth (anti-drift)

The builder hand-copies runtime facts in three places (DEFAULT_TOOL_NAMES, PROMPT_REGISTRY, the model/agent/profile dropdowns), and the tool list has already drifted. Fix the drift, then install a mechanism so it cannot silently drift again.

2.1 Align the tool catalog with the runtime's real defaults DriftQuick DoneB (+ A minor)

The runtime's isToolVisible() (src/runner/tool-profiles.js) exposes 16 tools by default — everything except apply_patch, bash, manage_shell_jobs, and lsp_query. The builder's catalog lists only 13, omits ask_user_question entirely, and renders enter_worktree/exit_worktree as unchecked even though they are on by default. Consequence: the moment a user touches any tool checkbox, the builder emits an explicit --tools list that silently removes the model's ability to ask clarifying questions — exactly the safety net a novice most needs.

Fix: add ask_user_question, enter_worktree, and exit_worktree to the catalog and DEFAULT_TOOL_NAMES, drive checkbox defaults from the same "everything-except-these-four" rule the runtime uses, and reconcile the HTML checked attributes with the JS list (A's minor finding: list_worktrees and run_skill currently disagree and are only papered over by the on-load style reset).

Done when: leaving the checkboxes untouched and customizing-then-restoring them both produce the runtime's real 16-tool default.

2.2 Replace "keep in sync" comments with a mechanism DriftMedium PartialA + B

Both assessments converge on the same principle: comments are promises, checks are mechanisms. Two complementary options, cheapest first:

(a) Drift test (start here): one node:test file that imports the runtime's default-visible tool set, model default, and template facts, then parses the builder's mirrored constants out of the HTML and asserts they match. Runs with the normal node --require ./test/setup.js --test test/runner/*.test.js flow and fails loudly when the runtime changes.

(b) Generated manifest (later, if drift keeps hurting): a small scripts/export-builder-manifest.js that dumps models, default tools, template frontmatter, agents, and profiles into a JSON block embedded in the HTML, plus a doc check that fails when the snapshot is stale.

Done when: changing a runtime default without updating the builder makes a check fail.

Phase 3 — Highest beginner payoff

With the page correct and honest, teach. These items turn the builder from a flag assembler into a guide, in line with this repo's novice-first rules ("define jargon once", "every command says where to run it and what success looks like").

3.1 Live plain-language "What this command will do" summary UXMedium DoneA + B

Render a one-to-three-sentence narrative next to the generated command, derived from the same state that builds it: "The runner will work in ⟨folder⟩, may read and edit files but will ask you in Terminal before each change, cannot run shell commands, will stop after 16 steps, and will save a transcript to ~/.bridge-runner/logs/." This translates flags into consequences — the blast radius a beginner actually needs — and catches misconfigurations ("wait, why does it say shell is on?") before they reach Terminal.

Done when: every permission style, shell state, and step/budget limit is reflected in the sentence as options change.

3.2 Plain-language labels first, flags second — plus a glossary UXMedium PartialA + B

The permission-style dropdown already proves the right pattern ("Look only — read files, no writes"). Apply it everywhere: "Auto-run tests after edits --test-watch" instead of "--test-watch (auto-run tests after writes)". Add a small glossary (inline <details> or "?" tooltips) defining each scary term once, in one sentence: worktree, ledger, compaction, fork, transcript, trace, session, fingerprint, chaos-ok, LSP.

Done when: no control is labeled with a bare flag, and every jargon term links to or shows a one-line definition.

3.3 Validate inputs and block obviously broken commands UXMedium DoneA

The chaos gate proves the pattern works — extend it. Warn (and soften the copy button) when: the prompt is empty; the target folder is not an absolute path; --test-watch is set without shell; replay/repair is checked without a session id/path; or --include-claude-md is set without --include-repo-context (a real runtime dependency the builder currently ignores).

Done when: each broken combination shows an inline explanation instead of producing a command that fails in Terminal.

3.4 "Before you run" preflight checklist with a copyable health check UXQuick DoneA + B

A small card near the copy button: (1) bridge running in VS Code — copy curl -s http://127.0.0.1:11437/health to verify in Terminal; (2) you are in the runner folder; (3) the target project folder is the one you mean; (4) first runs in a new folder will show the --trust-workspace consent step; (5) new here — start with a read-only preset. Each item says where to do it (Terminal vs VS Code) and what success looks like, removing the most common first-run failure (bridge not up) before it happens.

Done when: a first-time user can go from a fresh page to a successful run without opening --help.

3.5 Contrast and non-color status cues UXQuick PartialB

Nudge the smallest hint text up from 0.72rem, lighten --muted where it carries real guidance, and pair every color-coded warning with an icon or word ("Blocked", "Caution") so meaning survives for color-blind users and low-quality screens. Low effort, broad benefit.

Done when: every warning state is identifiable with color removed.

Phase 4 — Structure and durability

Larger reshaping of the page once the fundamentals are solid. These change layout and interaction patterns, so they benefit from the drift test (2.2) already being in place.

4.1 Split "Advanced" into goal-oriented tiers UXMedium PartialA + B

Replace the single ~70-control Advanced drawer with progressive disclosure. Merging both proposals: "Common extras" (attach files, stream, verbose, output format, human log), "Tuning" (model, tokens, steps, templates, budgets), and "Expert / recovery" (sessions, ledger replay/repair, trust flags, trace levels, custom system prompts). Each tier gets a one-line "use this when…" intro so a beginner scans by question, not by flag — three labeled shelves instead of one scary drawer.

Done when: a beginner can tell at a glance that --stream and --repair are not the same class of control.

4.2 Beginner mode toggle, defaulted ON UXBigger PlannedB

A top-level switch that collapses the page to roughly eight controls: prompt, presets, permission style, the two folders, and the command/summary panel. Everything else stays one click away. Best done after 4.1 so the tiers give the toggle something clean to hide.

Done when: the default first-visit view fits comfortably on one screen with no jargon visible.

4.3 "Explain this command" breakdown + save/restore/reset UXBigger PartialA

Two learning aids: (a) an expandable per-flag breakdown under the terminal box — each emitted flag as a row with a one-line meaning and a risk badge (read / write / shell / recovery), turning the command into a vocabulary lesson; and (b) persist the form to localStorage with a "Reset to defaults" button (and optional URL-hash sharing), so a page refresh doesn't destroy ten minutes of careful configuration — the most demoralizing possible beginner experience.

Status: Part (b) — localStorage persistence and "Reset to defaults" — is done. Part (a) — per-flag breakdown with risk badges — is still planned.

Done when: refreshing the page restores the previous configuration, and every flag in the output has a visible one-line meaning.

4.4 Explain the form's automatic changes UXMedium PlannedB

When enabling shell auto-checks bash, or a model choice triggers a temperature warning, show a tiny inline note explaining why a box just changed. Today the form mutates itself silently, which is confusing when you're still learning cause and effect.

Done when: every automatic state change is accompanied by a short visible explanation.

Backlog / nice-to-have

  • "Handy commands" discovery box (A + B, low severity): a short card listing --list-agents, --list-profiles, and the runner eval / runner worktrees list subcommands the builder intentionally doesn't generate, so beginners can discover them without reading --help.
  • Model list freshness pointer: until 2.2(b) exists, add a note next to the model dropdown explaining how to check which models the bridge actually accepts, so a silent 400 has a visible cause.

Suggested sequencing at a glance

Batch Items Why this order
Now 1.1, 1.2, 1.3, 1.4, 2.1 All quick, all correctness. After this batch the page renders as designed, every preset works as promised, quoting is safe, and the tool panel stops lying about defaults.
Next 2.2(a), 3.1, 3.4 Install the drift test before touching more mirrored data, then land the two highest-payoff teaching features: the "what will happen" summary and the preflight checklist.
Then 3.2, 3.3, 3.5 Labels, glossary, validation, and accessibility — steady beginner-experience improvements.
Later 4.1, 4.2, 4.3, 4.4, 2.2(b), backlog Structural redesign and durability, done on a page that is already correct and honest.

Everything in this roadmap is scoped to docs/command-builder.html, one optional test file, and one optional script. No runner or bridge behavior changes are required, so none of the transport/auth invariants are touched.

Checks to run when items land

  • npm run check:docs — validates the docs after HTML edits.
  • npm run lint and npm run format:check — before handoff.
  • Once 2.2(a) exists: node --require ./test/setup.js --test test/runner/*.test.js to confirm the builder still matches the runtime.
  • Manual: open the builder in a browser, click each preset, and read the generated command aloud against the "done when" line of the item you just fixed.
