Runner Expansion Roadmap — Critical Assessment

A second opinion on docs/runner-expansion-roadmap.md, plus five future directions that the current document either under-weights or omits entirely.

companion to runner-expansion-roadmap.md · v1 · read-only assessment

Part 1 — Honest assessment of the existing roadmap

What the roadmap gets right. The document is unusually thoughtful for a roadmap. It avoids vendor lock-in, anchors every proposed feature to the threat model (read-only / writes / shell / orchestration categories), is realistic about what is already shipped (test-watcher, presets, agent profiles, structured tools), and consistently steers toward "docs / presets / agents" rather than reflexively adding tools. The "Principle to encode in presets" sidebar (every implement preset nudges toward a verifier) is a strong piece of process design, not just feature design.

Five honest gaps

  1. The roadmap is feature-shaped, not failure-shaped. It enumerates capabilities to add (run-log search, test-watcher, repo map, prompt library) but contains very little material on regression budgets, observability of the runner itself, or how to know a release broke something subtly. There is a threat model but no operability model.
  2. "Spawn" is treated as one feature, not a system. spawn_agent ships with a fork-depth=1 cap and an 8-spawn-per-run cap (see src/runner/tools/spawn-agent.js), but the roadmap does not discuss how a richer orchestration story (queues, retries, parallelism, partial-result merging, cancellation, cost ceilings) would interact with those invariants. Today the cap is a hard ceiling; tomorrow it will be a UX limitation that users route around by chaining shell calls.
  3. Evaluation is mentioned, not designed. The roadmap promises "verification presets" and "tests after edits", which are tactics. There is no evals harness for the runner itself — a way to replay a known-good transcript and assert that a new build produces equivalent tool calls, permission decisions, and trace shape. Without this, every refactor of tool-catalog or worker-runtime is a leap of faith.
  4. The command-builder is positioned as docs, but it is software. It contains real parsing logic (shell-quoting, session-mode conflict detection, chaos guards). A doc-style page accreting JS becomes a maintenance liability — a stress test of the current file (see Part 3 of the parent conversation) surfaced a CSS bug that has been shipped, a shell-quoting bug that is exploitable on attacker-controlled filenames, and several state-machine inconsistencies. The roadmap should call out a plan for either slimming the builder or adopting tests for it.
  5. Recovery is under-specified. The roadmap mentions replay/repair and traces, but a "you ran the wrong command — what now?" story is missing. Backups, undo, and recovery tools are listed under capability groups but never as workflows. A user who has just had a botched --accept-edits run wants a single cc undo last-run, not a five-step grep through transcripts.

Where the roadmap could be braver

  • It catalogs Anthropic's published tips at length (the long table) but rarely says "we explicitly disagree with this and here is why." A roadmap that says no out loud is a stronger roadmap. (Examples: subagents-for-context-windowing is presented neutrally, but the threat model arguably implies the runner should resist long-running parallel agents on local machines.)
  • "Docs/presets" is doing a lot of work as a bucket. Roughly a third of the table maps to it. Some of those rows deserve to be their own features (e.g. a real prompt-template registry under .bridge-runner/prompts/, with discovery and validation, not just "write a markdown file").

Part 2 — Five additional future directions

Each direction below names an explicit gap in the current roadmap, proposes a minimum-viable shape, and lists what it would force us to decide. Effort labels are rough order of magnitude. Categories align with the existing roadmap's framing (Safety, Developer experience, Operations, Evals).

1 Runner self-evals: golden-transcript replay harness

Treat the runner like a compiler — assert that a new build produces the same tool calls, permission decisions, and trace shape on a corpus of canned sessions.

Category: Evals Effort: Medium Phase 2 No new tools

Why

Every refactor of tool-catalog.js, permissions.js, or worker-runtime.js can change the shape of tool calls, the order of permission checks, or the bytes that flow to disk in a trace. There is currently no way to know whether such a change is a bugfix or a regression — the only signal is an angry user a week later. This is the largest silent risk in the project.

Minimum-viable shape

  • Add test/runner/golden/ with canned transcripts: for each, a pinned model output stream (the assistant's tool-call JSON) and the expected runner side of the interaction — which tools were dispatched, with what permissions, what files were touched, what bytes were written to the trace.
  • A cc runner eval command that replays the model stream into the runner with a fake model client, then diffs actual vs. expected runner behavior.
  • CI gate: PRs to src/runner/ must not regress goldens without an explicit "approve regression" commit message.

What it forces us to decide

  • What counts as "behavior" — exit code only, or full tool-call sequence, or trace bytes?
  • How to make goldens portable when paths/timestamps differ.
  • Whether to redact secrets from goldens (yes) and how that interacts with full-trace mode.

2 First-class recovery workflow: cc undo last-run

Promote backups and per-run history from "tools that exist" to a single, discoverable workflow. The runner already has the primitives; users just don't know how to chain them.

Category: Safety Effort: Small Phase 1 CLI + docs

Why

Today the recovery story is: "look at .bridge-runner/backups/, find the file, run undo <path>". This is fine if one file was touched. It is hostile if a run with --accept-edits rewrote a dozen files and the user wants to revert the entire session. The runner already records which files were edited (it has to, for the backup system) — it just doesn't expose that record as a unit.

Minimum-viable shape

  • Per-run manifest: every session writes .bridge-runner/runs/<session-id>/manifest.json with the list of edits made and which backup file restores each.
  • cc undo last-run reverts every file in the most recent run's manifest, in reverse edit order, with a diff preview and confirmation prompt.
  • cc undo run <session-id> and cc undo list-runs for older sessions.
  • The command-builder grows a "recovery" tab pointing at these commands so they are discoverable without reading the README.

What it forces us to decide

  • How to handle partial reverts when a later run also touched a file.
  • Whether to store inverse-diff patches (small, robust) or full backups (large, simpler).
  • Garbage-collection policy for old run manifests.

3 Cost & budget telemetry as a first-class signal

Make tokens, wall-clock, and spawn-count visible during a run and enforceable as soft and hard caps. Currently the only cap is the per-run spawn ceiling.

Category: Operations Effort: Medium Phase 2 CLI + runtime

Why

Users currently learn that a run was expensive after the bill arrives. The roadmap mentions usage counters in traces but no workflow around them. Three concrete pain points: (a) no way to say "stop the run if input tokens exceed X"; (b) no live signal during long sessions; (c) child agents from spawn_agent have their own budgets that the parent cannot observe.

Minimum-viable shape

  • Three new flags: --budget-input-tokens N, --budget-output-tokens N, --budget-wall-seconds N. Soft caps emit a structured warning event; hard caps end the run cleanly with a "budget exceeded" state.
  • A new event in the trace stream: { type: "budget", input_tokens, output_tokens, wall_ms, spawns, depth }, emitted at every tool boundary. Trace consumers (incl. flight recorder) can plot this.
  • Child agents inherit the parent's remaining budget by default; a future spawn_agent argument can carve out a sub-budget explicitly.
  • The command-builder grows a "budget" panel in the Permissions section.

What it forces us to decide

  • Whose token count is authoritative — API response or local estimate.
  • How to surface a soft-cap warning to the model without confusing it.
  • Whether hard-cap termination unwinds in-flight edits.

4 Tool capability profiles & per-tool deny-by-default

Move from the four coarse permission categories (read / writes / shell / orchestration) toward composable per-tool profiles, while keeping the simple categories as the default UX.

Category: Safety Effort: Large Phase 3 Runtime + tools + docs

Why

Today, --accept-edits permits all write tools, and --allow-shell permits bash wholesale. Users frequently want something narrower: "allow edit_file but not write_file", "allow bash but only for git status", "allow spawn_agent but cap depth at 1 even for trusted roots". Today this requires either accepting more than necessary or writing custom agent profiles.

Minimum-viable shape

  • Tool profiles in .bridge-runner/profiles/<name>.json describing per-tool allow/deny rules, optional arg-shape constraints (regex on shell command, max file size for write_file, etc.), and a human-readable rationale.
  • --profile name CLI flag layered over the existing category flags, so the simple story still works.
  • Profiles are first-class in the command-builder as a dropdown alongside the current permission style.
  • A built-in profile per common scenario: review-only, edit-source-no-shell, git-readonly-shell, etc.

What it forces us to decide

  • Profile semantics: deny-overrides-allow vs. last-match-wins.
  • How profiles compose with the chaos-mode (all-three-on) interlock — profiles must not bypass the interlock.
  • Whether the model is told about its profile (yes — denied tools should be visibly absent or marked).

5 A prompt-template registry under .bridge-runner/prompts/

Stop treating prompt templates as folklore. Give them discovery, validation, parameters, and provenance — the same things tools and agents already have.

Category: DX Effort: Small / Medium Phase 1–2 Files + CLI

Why

The existing roadmap calls out grill, simplify, verify, explanatory, etc., as "docs/presets" entries — i.e. write a markdown file somewhere. That works at small scale and falls over at medium scale: there is no listing, no parameter validation ("this template expects a $path variable"), no version/owner metadata, no way for one project to inherit templates from a parent project.

Minimum-viable shape

  • .bridge-runner/prompts/<name>.md with YAML frontmatter: title, summary, parameters: [...], recommended-tools: [...], recommended-permissions: ..., tags: [...].
  • CLI: cc prompts list, cc prompts show <name>, cc prompts validate.
  • --prompt-template name already exists; extend it with --prompt-arg key=value for parameter substitution at runtime.
  • The command-builder learns to read this registry (file:// fetch or shipped JSON) and auto-suggest the recommended permissions and tools for the chosen template — closing the loop between prompt intent and runtime safety.
  • Built-in templates ship in src/runner/prompts/ and user templates override them by name.

What it forces us to decide

  • How to handle prompt-injection from user-supplied parameter values (escape, or refuse strings that look like control tokens).
  • Whether templates can compose ("verify-then-grill") or stay flat.
  • The provenance/trust story for templates pulled from elsewhere.

Part 3 — At-a-glance summary

# Direction Category Effort Phase Closes which gap?
1 Golden-transcript replay harness Evals Medium 2 Silent regressions in runner refactors
2 cc undo last-run recovery workflow Safety Small 1 Recovery is primitive-only, no workflow
3 Budget telemetry & caps (tokens, wall, spawns) Operations Medium 2 Cost is invisible until it isn't
4 Composable tool capability profiles Safety Large 3 Permission categories are too coarse
5 Prompt-template registry with frontmatter DX Small/Med 1–2 Prompt templates are folklore, not artifacts
Sequencing recommendation. Start with #2 (recovery) and #5 (prompt registry) — both are small, both unlock UX wins, and neither perturbs the runtime. Then #1 (golden-transcript evals) before any work on #3 or #4, because both #3 and #4 will refactor permission and runtime paths, and you want a safety net first.

Part 4 — Explicit non-goals

A roadmap that says no is more useful than one that says maybe. These ideas were considered for inclusion and explicitly dropped:

  • Removing the spawn_depth = 1 cap. Tempting (it would enable richer orchestration), but the threat model relies on a finite tree. Solve queueing and budgets first (#3), then revisit.
  • A web UI that runs the runner remotely. Diverges from the "local bridge" identity. The command-builder is the right level of UI.
  • A plugin marketplace for tools. Tools are the most security-sensitive surface in the system. Stay curated.
  • Auto-applying model-suggested permission escalations. If the model asks for shell access mid-run, the answer is no. The user re-runs with broader permissions; the runner does not negotiate.
