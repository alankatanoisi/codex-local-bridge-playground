Codex Bridge Runner — Feasibility & Roadmap

Can the Claude Code (“cc”) bridge runner architecture be reused to build a Codex local bridge runner that drives OpenAI's Codex models through the same local agent loop? Short answer: yes — this document reviews the current runtime capabilities, measures the provider coupling, and lays out a phased build plan.

TL;DR verdict

✅ Feasible — roughly 85% of the runner carries over as-is

The runner was built with a clean seam between “talk to the model” and “everything else” (tools, safety, permissions, sessions, archives, prompts, profiles, hooks, undo, coordinator). The “everything else” layer never looks at Anthropic-specific data shapes — it consumes a neutral { name, input, id } tool call and produces plain text results. Only four core modules speak the Anthropic Messages dialect natively, and each has a well-understood OpenAI Responses-API equivalent.

Decision (2026-07-08): Alan chose Option C — fork the playground into a separate codex-local-bridge-playground repository and rewrite the model-facing modules in place there. Auth uses an official ChatGPT Business programmatic access token (at-…, scoped to Codex / Workspace Agents) supplied via an environment variable, which removes both the OAuth phone-verification blocker and the token-refresh machinery. The roadmap in Part 5 is written for this fork-and-prune path: 6 phases, with a usable proof-of-concept Codex runner at the end of Phase 4.

⚠️ Two caveats before anything is built.

1. Policy risk (now much smaller than first assessed). The original concern was reusing ChatGPT OAuth credentials outside the official Codex CLI. The chosen credential is instead an official ChatGPT Business access token, created in the dashboard for “ChatGPT and Codex programmatic use cases” — a sanctioned programmatic path. Residual risk is narrower: confirm in Phase 0 what the token is documented to permit, and keep the personal-research framing until that is pinned down.

2. Wire-format facts need re-verification. The Codex CLI and its backend endpoints evolve quickly. Every protocol detail in Part 3 is best current knowledge and is re-verified as the very first roadmap task (Phase 0) against the open-source Codex CLI code before any adapter code is written.

Part 2 — Where the Anthropic-specific code lives

A grep for Anthropic message-shape vocabulary (tool_use, tool_result, content_block, input_schema, cache_control, thinking) finds 121 hits across 39 runner files — but the distribution is very lopsided. Most files have exactly one hit: the input_schema key in their tool definition, which is a mechanical one-line concern. The real coupling is concentrated in four modules:

ModuleAnthropic-specific behaviorCodex-side equivalentPort difficulty
src/runner/model-client.js (262 lines) Messages API request/response; SSE event grammar (message_start, content_block_delta, thinking_delta/signature_delta, input_json_delta) Responses API request/response; SSE grammar (response.output_text.delta, response.function_call_arguments.delta, response.completed) Medium — write a sibling client, same size
src/runner/run.js (1,370 lines) Content-block extraction; cache_control breakpoint budgeting (4-marker budget with bridge reserve); thinking-block + signature preservation across turns; usage field names No manual cache markers (OpenAI caching is automatic); reasoning items with encrypted_content must be replayed — structurally the same “preserve opaque block” problem the runner already solves for thinking signatures Medium — mostly skipping logic per provider, not adding it
src/runner/tool-pipeline.js (532 lines) Assembles tool_use / tool_result message pairs and ids Assembles function_call / function_call_output items keyed by call_id Low — if the adapter translates at the client boundary, this file does not change at all
src/runner/context-compactor.js (337 lines) Compaction ladder walks Anthropic message/content-block shapes Same, if translation happens at the boundary Low — unchanged under the boundary-adapter design
Tool definitions (24 files under src/runner/tools/) input_schema key (Anthropic name for the JSON Schema) parameters key (OpenAI name for the same JSON Schema), optional strict: true Low — one shared mapping function in the adapter; tool files untouched
Key architectural insight.

Because tool-pipeline.js, context-compactor.js, transcripts, the archive, and the golden-eval harness all speak the internal Anthropic-block dialect, the cheapest correct design is to keep that dialect as the runner's internal lingua franca and translate only at the model-client boundary: Codex function_call items become internal tool_use blocks on the way in, and internal tool_result blocks become function_call_output items on the way out. One adapter file, everything downstream unchanged.

Part 3 — What Codex looks like on the wire

Codex CLI
OpenAI's open-source local coding agent (github.com/openai/codex, Apache-2.0, written in Rust). It is the OpenAI counterpart of Claude Code.
Responses API
OpenAI's current model API. Where Anthropic sends a messages array plus a system prompt, OpenAI sends an input item list plus an instructions string.
ChatGPT OAuth
Signing in to Codex CLI with a ChatGPT account instead of an API key. Tokens are stored locally in ~/.codex/auth.json (access token, refresh token, account id) — the direct analog of Claude Code's keychain entry / ~/.claude/.credentials.json. Not usable here: the OAuth login flow sends a phone OTP to a number no longer owned.
Programmatic access token (at-…)
A token created in the ChatGPT Business dashboard under “Access tokens — create and manage access tokens for ChatGPT and Codex programmatic use cases”, scoped (in this case) to Codex and Workspace Agents. Dashboard-managed: revoke and re-create there in seconds, no client-side refresh flow. This is the chosen credential for the fork.
SSE (server-sent events)
The streaming format both providers use: a long-lived HTTP response that emits small JSON frames as the model generates.

The structural takeaway: every Anthropic concept the runner relies on has a direct Codex counterpart, and the two hardest problems the runner already solved — streamed partial tool-call JSON assembly and opaque reasoning-block round-tripping — are the same problems in slightly different clothes. Part 4 — Architecture decision: three ways to build it

The playground has a hard transport invariant: the Claude bridge stays Anthropic-native (no /v1/chat/completions, no OpenAI routes, no API-key fallback). So “add OpenAI support to the existing bridge” is off the table by design. That leaves three shapes:

Option A — Standalone Codex credential daemon + runner provider adapter Not chosen

A new, small, separate local daemon (a plain Node script — no VS Code extension needed, since Codex credentials live in a readable file rather than behind editor interception) listens on its own port (e.g. localhost:11438), discovers ChatGPT OAuth from ~/.codex/auth.json / env var, handles token refresh, and forwards native Responses-API requests. The runner gains a --provider codex flag that swaps in a Codex model client whose adapter translates to the internal Anthropic-block dialect at the boundary.

  • Pros: preserves the bridge/runner separation the whole repo is built on; credentials stay out of the runner process; keeps the trace-correlation, caller-auth, and redaction patterns; Claude bridge invariants untouched; the daemon is reusable by other local clients later.
  • Cons: one more process to start; small new codebase to maintain (~300–500 lines).

Option B — Provider adapter only, runner talks to OpenAI directly

  • Pros: no new daemon; fastest to a demo.
  • Cons: moves credential discovery, refresh, and redaction into the runner, breaking the “bridge owns auth, runner owns the loop” boundary; loses trace correlation and caller auth; makes the runner itself hold live account tokens, which the threat model currently avoids.

Option C — Fork the repo into a separate codex-local-bridge project Chosen

  • Pros: zero risk to the Claude runner; total freedom.
  • Cons: permanent double maintenance of ~10,000 lines of runner code; every safety fix must be applied twice; directly contradicts the “one runner, small extension points” direction of this lab.
Decision: Option C, recorded 2026-07-08.

Alan chose the fork for proof-of-concept freedom: the new repo can delete Claude-specific transport code outright instead of abstracting around it, and its agent instructions (CLAUDE.md / AGENTS.md) can be written for OpenAI work without conflicting with this repo's Anthropic-only invariants. The known cost — double maintenance of shared runner code — is accepted for the PoC stage; if the fork proves out and porting fixes by hand becomes painful, restructuring into a shared-core monorepo is the natural upgrade path. Target repo: alankatanoisi/codex-local-bridge-playground (a separate repository, not a folder inside this one, so the two projects' agent rules, credentials, and histories never mix).

Part 5 — Phased build roadmap

Rewritten for the chosen Option C fork. All build phases from Phase 1 onward happen in the new codex-local-bridge-playground repository; this playground repo is only the seed source and keeps this document as the origin spec (a copy should travel with the fork and become its living roadmap). “Session” means one focused agent working session. Exit criteria are deliberately test-shaped because the golden-eval harness lets most of this be verified with no live credentials at all.

Phase 0 — Protocol & token spike 1–2 sessions

runs anywhere · new: lab-notes doc · touches: no src/ in either repo
  • Verify what the at-… Business access token can call: exact endpoint URL(s), required headers (Bearer plus any workspace/account id), and whether it speaks the standard Responses API shape. Pin this from OpenAI's access-token docs and the open-source Codex CLI source, then prove it with one redacted curl capture (one-shot and streamed) kept as the reference fixture.
  • Confirm the token's scopes (“Codex”, “Workspace Agents”) cover the calls the runner needs, and record its expiry/rotation story (dashboard-managed; expected: revoke and re-create, no client-side refresh flow).
  • Write down what OpenAI documents the token as permitted for, so remaining policy risk is recorded fact rather than guesswork.
  • Decide the Codex model list and pricing rows for model-pricing.js.

Exit criteria: every row of the Part 3 table above is confirmed or corrected with citations; a scratch script completes a live streamed Responses call using the env-var token; go/no-go checkpoint with Alan.

Phase 1 — Create and seed the fork 1–2 sessions

new repo: alankatanoisi/codex-local-bridge-playground · this repo: untouched
  • Create the empty GitHub repo (private recommended for a PoC that handles credential-adjacent code), copy this playground's files in, and make one “seed” commit that records provenance (source repo + commit hash) so future diffs against the Claude runner stay possible.
  • Prune (delete): the VS Code extension surface and Claude bridge internals (src/server.js, src/proxy.js, src/credentials.js, src/interceptors/**) and their tests, Claude-specific docs lanes and archived handoffs, and vendored Claude agent collections.
  • Keep: bin/, src/runner/**, test/runner/**, scripts/, lint/format configs, docs skeleton.
  • Rewrite: README, CLAUDE.md/AGENTS.md (a new lane table so agents never confuse the two repos, plus the fork's transport invariants: upstream auth is the at-… token from one env var only; never commit, print, or log it; redact at-/sk-/eyJ patterns everywhere; native Responses route only; no Anthropic credentials or routes in that repo), and package.json name/metadata.

Exit criteria: fork pushed; after pruning, npm test and lint are green in the fork (runner tests only — the model client is still the Claude one at this point, which is fine because golden evals use a fake client).

Phase 2 — Transport: direct token client (daemon optional later) 1–2 sessions

fork repo · new: transport module · touches: safety.js redaction patterns
  • With a dashboard-managed static token there is no keychain to read, no interception, and no refresh flow — a separate credential daemon adds little for a PoC. Start direct: the forked runner's model client calls the OpenAI endpoint itself, reading the token from one env var (e.g. CODEX_ACCESS_TOKEN). Keep “split transport into a small local daemon on its own port” as a later refinement if multi-client use or request interception ever matters.
  • Before the first live call from runner code: extend secret redaction to at-… token shapes, verify the deny matrix blocks any file the token could live in (.env is already covered), and confirm subprocess environment scrubbing strips the token from child shells.
  • Port the flight-recorder trace pattern so request boundaries stay auditable with header names, never header values.

Exit criteria: transport unit tests with fixture tokens prove redaction and env scrubbing; a live smoke script round-trips one-shot and streamed calls through the new transport module.

Phase 3 — Rewrite the coupled modules in place 3–4 sessions

fork repo · touches: model-client.js, run.js, model-pricing.js · unchanged: tool-pipeline.js, context-compactor.js, all 24 tools
  • model-client.js becomes a Responses API client: system prompt → instructions; tool definitions → { type: "function", …, parameters } via one shared input_schema → parameters mapping (tool files untouched); internal messages → input items; --effort → reasoning.effort; store: false plus reasoning include for stateless multi-turn.
  • Keep the internal Anthropic-block dialect as the fork's internal representation — the Part 2 insight applies inside the fork too. function_call items translate to internal tool_use blocks at the client boundary, so the tool pipeline, compactor, transcripts, archive, and golden evals stay byte-compatible.
  • SSE mapper: Responses event grammar → the existing streamed-block assembly (the partial tool-argument buffering in postStream carries over almost verbatim); reasoning items ride the opaque-preserved-block path built for thinking signatures.
  • run.js: delete cache_control budgeting entirely (OpenAI caching is automatic); usage normalizer maps cached_tokens → cache_read_input_tokens with cache-write fixed at 0; add Codex rows to the pricing table.

Exit criteria: unit tests translate the Phase 0 captured fixtures in both directions; at least one golden-eval case drives the full loop against a scripted fake Codex client.

Phase 4 — Live proof-of-concept runs 1–2 sessions

fork repo · throwaway labs only · docs: fork quickstart + command builder
  • Climb the capability ladder in a throwaway lab, in this order: read-only run → --plan → write tools with confirmation → --accept-edits → --allow-shell → spawn_agent → worktrees → --stream.
  • Confirm budgets, the cost summary line, transcripts, archive, human log, undo manifests, and traces all behave with Codex usage numbers.
  • Tune the default system prompt if Codex models respond differently to it — data-driven, using the archive to compare runs.

Exit criteria: the throwaway-lab smoke task (“fix the calculator bug, run npm test, summarize”) passes end-to-end on a Codex model.

Phase 5 — PoC hardening & docs 1–2 sessions

fork repo · touches: threat model, README, golden cases
  • Golden-eval cases for permission denials, budget stops, repeat-tool detection, and compaction under the Codex client.
  • Close the gaps found live in Phase 4: multimodal (input_image; if PDF is unsupported, degrade to text extraction with a clear tool note), retry/backoff tuning, token-count preflight differences.
  • Fork threat-model doc: the at-… token class, env-var handling, redaction coverage, and the Phase 0 policy findings.
  • Start a PORTING.md log in the fork: every safety or runner fix cherry-picked from the Claude playground gets one line, so divergence stays visible instead of silent.

Exit criteria: the fork's npm test, lint, and doc checks are green; a written “known gaps vs the Claude runner” list exists.

Phase 6 — Optional: cross-runner experiments exploratory

  • Run the same task on both runners and compare archives: cost, steps, stop reasons, tool-call patterns.
  • Cross-runner A/B on the Harbor smoke tasks with identical prompts.
  • Longer term: if the two-repo split proves annoying, this is where a shared-core monorepo restructure (Part 4's Option A shape) gets reconsidered with real evidence in hand.

Exit criteria: none — this is the payoff playground once Phases 0–5 are stable.
