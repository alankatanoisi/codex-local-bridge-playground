Command Builder — Critical Assessment

A review of docs/command-builder.html against the current runner runtime (bin/local-bridge-runner.js and src/runner/**), focused on one question: is this page actually optimized for a beginner programmer?

Command Builder Runner Quick Start

Overall verdict

The command builder is feature-complete but beginner-hostile in its current shape. It faithfully mirrors nearly the entire runner flag surface (roughly 60 CLI flags), which makes it a great reference tool — but completeness is the enemy of a first-time user. The page front-loads safety (presets, permission styles, warning panels, the chaos gate), which is genuinely good design. But once the Advanced panel opens, a beginner faces ~70 controls with raw flag names as labels, jargon with no definitions (ledger, worktree, compaction, fork, trace, session extract), and no narrative explanation of what the generated command will actually do. It also contains at least three real bugs, including a broken CSS selector that silently disables the intended two-column layout.

What it gets right strengths

  • Safety-first ordering. Prompt → presets → permission style is the correct funnel. The five-option "What may the runner do?" dropdown is the single best beginner element on the page — it converts four raw flags (--plan, --accept-edits, --dont-ask, --allow-shell) into one plain-language decision.
  • The chaos gate works. Shell + auto-edits + no-prompts is blocked in the UI (copy button disabled) exactly like the runner blocks it without --chaos-ok. UI policy matches runtime policy.
  • Live caution panels. The dynamic warnings (accept-edits skips confirmations, dont-ask does not enable bash, trace-level full writes prompts to disk, ack-resume-risk overrides the health gate) mirror the real safety model in src/runner/safety.js and the CLI help text.
  • Flag coverage is accurate. Spot-checking every emitted flag against parseArgs in bin/local-bridge-runner.js: all builder flags exist in the runner, defaults match (claude-sonnet-4-6, max-tokens 2000, max-steps 16, shell-timeout 30000), and default-suppression (only emit non-default values) keeps commands short.
  • The recovery card is excellent beginner material. Showing local-bridge-undo.js with a dry-run-first sequence teaches the right habit: preview, then act.
  • Presets seed a real prompt, not just flags — a beginner sees a complete working example.

Bugs found during review defects

These are concrete defects in the current file, independent of any beginner-experience opinion. Line numbers refer to docs/command-builder.html as of this review.

bug 1The .layout CSS selector is missing — the two-column layout never applies

Around lines 68–73 the stylesheet contains orphaned declarations (display: grid; grid-template-columns: 1fr 1fr; …) with no selector in front of them — the .layout { line was lost in an earlier edit. Browsers silently drop the invalid block, so the page renders as one long single column and the "sticky command panel" on the right never behaves as designed. The @media (max-width: 700px) fallback right below it still references .layout, confirming the selector was intended to exist.

bug 2The "Verify after edits" preset silently drops --allow-shell and --test-watch

The verify preset declares allowShell: true, testWatch: true, but applyPreset() only applies the full boolean key list when the preset has no permissionStyle. Because verify sets permissionStyle: 'edit-auto', only the whitelist (stream, verbose, bareContext, taskScope, newSession, resumeSession) is applied — so allowShell and testWatch are ignored. The preset's own prompt says "let --test-watch surface test results", but the generated command contains neither flag, and --test-watch requires --allow-shell at runtime anyway. A beginner clicking this preset gets a command that cannot do what its prompt promises.

bug 3--session-id / --session-path are emitted twice in replay/repair mode

buildCommandParts() pushes --session-id and --session-path unconditionally near the top, then pushes them again inside the replayMode / repairMode branches before returning early. With a session id set and replay checked, the copied command contains the flag twice. Node's parseArgs takes the last value so it happens to work, but a duplicated flag in a teaching tool erodes trust ("did I do something wrong?").

minorSmaller correctness nits

  • HTML checkbox defaults disagree with DEFAULT_TOOL_NAMES. list_worktrees and run_skill are in the JS default list but unchecked in the HTML. It is papered over because applyPermissionStyle('edit-ask') runs on load and resets the boxes, but any future refactor of that init call resurfaces a spurious --tools flag on the default command.
  • shellQuote() is unsafe for mixed quoting. A prompt containing a single quote is switched to double-quote wrapping, but $, backticks, and \ are not escaped — a prompt like don't touch $HOME produces a command where the shell expands $HOME before the runner ever sees it. For a beginner pasting free-text prompts, silent shell expansion is exactly the kind of confusing failure the builder exists to prevent.
  • An empty prompt still produces a copyable command. Nothing warns that the runner will be invoked with no positional prompt.
  • Hard-coded model list is a drift magnet. The dropdown ships six model ids as static HTML with no pointer to how to check what the bridge actually accepts. Same drift risk applies to PROMPT_REGISTRY (a hand-copied snapshot of template frontmatter) and the agent/profile dropdowns.

Beginner-experience assessment critique

1. The Advanced panel is a cliff, not a ramp

The default view (prompt, presets, permission style, two folders) is well-scoped. But "Advanced options" is a single <details> hiding six dense cards and ~70 controls — attachments, model tuning, prompt templates, logging, session health, ledger repair, budgets, trust flags, and the full tool catalog — with no internal hierarchy of "you might want this soon" versus "you will likely never need this." A beginner who opens it once has no way to tell that --stream (harmless, useful) and --repair (mutates the session ledger) live at the same level of the interface.

2. Labels are flag names, not questions

Most checkboxes are labeled with the literal flag (--compact-each-turn, --exclude-dynamic-system-prompt-sections, --ack-resume-risk) and nothing else. The permission-style dropdown proves the page knows how to do better ("Look only — read files, no writes"). That pattern should be everywhere: plain-language label first, flag name second as a hint.

3. Jargon is used without definitions

The repo's own novice-first rules say "define jargon once when you use it." The builder uses worktree, ledger, compaction, fork, transcript, trace, session extraction, fingerprint, and LSP with zero definitions. A beginner cannot evaluate "Isolate edits on a git worktree branch" without knowing what a worktree is.

4. The output is a command, not an explanation

The right panel shows the assembled command string — which is the artifact, not the understanding. There is no "here is what will happen when you press Enter" narrative (which tools are exposed, whether you will be asked before edits, when the run stops, where the transcript lands). The runner's --help has a "Beginner notes" section; the builder, ironically, has less hand-holding than the raw CLI.

5. No preflight, no persistence, no recovery from mistakes in the form itself

The page assumes the bridge is already running and the folders are right, with only a buried tip about Unauthorized: Missing Bearer token. There is no "before you run" checklist, no way to save or share a configuration (a page refresh wipes everything), and no reset-to-defaults button after experimenting.

6. Guidance that exists is easy to miss

The best beginner advice on the page ("New here? Try a read-only preset first") is the fifth bullet of a muted-gray list under the prompt box. The riskiest control (--chaos-ok) is a plain checkbox in the Session & context card, physically distant from the permission card whose warnings reference it.

10 suggested improvements roadmap

Ordered roughly by impact-for-effort. Items 1–3 are bug fixes; 4–8 are beginner-experience redesigns; 9–10 are anti-drift and learning aids.

1Fix the missing .layout selector

Restore .layout { before the orphaned grid declarations so the intended two-column layout (form left, sticky command panel right) actually renders. This is a one-line fix with a large payoff: the command preview stays visible while a beginner changes options, which is the entire point of a live builder.

2Fix the "Verify after edits" preset so its flags actually apply

Either allow presets to layer extra booleans on top of a permission style (add allowShell and testWatch to the post-style whitelist in applyPreset()), or change the preset to use the edit-shell style. Also re-run updateUiState() ordering so the shell-dependent checkboxes don't immediately un-check testWatch. A preset that silently produces a command that can't do what its own prompt describes is worse than no preset.

3De-duplicate --session-id/--session-path and harden shellQuote()

Remove the duplicate pushes in the replay/repair branches, and make shellQuote() use proper POSIX single-quote escaping ('…'"'"'…' style) so prompts containing ', $, or backticks can never be expanded by the shell. Beginners paste free text; the quoting must be bulletproof.

4Split "Advanced" into three tiers of progressive disclosure

Replace the single Advanced panel with: "Common extras" (attach files, stream, verbose, output format, human log), "Tuning" (model, tokens, steps, templates, budgets), and "Expert / recovery" (sessions, ledger replay/repair, trust flags, trace levels, custom system prompts). Add a one-line "who needs this" sentence under each tier heading. The mental model shifts from "one scary drawer" to "three labeled shelves."

5Add a plain-language "What will happen" summary next to the command

Render a live narrative under the terminal box, derived from the same state that builds the command: "The runner will work in ⟨folder⟩, may read and edit files but will ask you in Terminal before each change, cannot run shell commands, will stop after 16 steps, and will save a transcript to ~/.bridge-runner/logs/." This turns the builder from a flag assembler into a teaching tool, and it catches misconfigurations ("wait, why does it say shell is on?") before they reach Terminal.

6Rewrite every control label as plain language first, flag second

Follow the permission-style dropdown's own pattern everywhere: "Auto-run tests after edits --test-watch" instead of "--test-watch (auto-run tests after writes)". Add a small glossary (<details> or hover definitions) for worktree, ledger, compaction, session, transcript, and trace — the repo's novice-first rules require defining jargon once, and this page currently never does.

7Add a "Before you run" preflight checklist with a copyable health check

A small card above the command: (1) bridge running in VS Code — copy curl -s http://127.0.0.1:11437/health to verify, (2) you are in the runner folder, (3) the target project folder is the one you mean, (4) for first runs in a new folder, expect the --trust-workspace consent step. Each item says where to do it (Terminal vs VS Code), matching the house rule that every command names its location and success criteria.

8Validate inputs and block obviously broken commands

Warn (and soften the copy button) when: the prompt is empty; the target folder is not an absolute path; --test-watch is set without shell; replay/repair is checked without a session id/path (the note exists but nothing enforces it); or --include-claude-md is set without --include-repo-context (a real runtime dependency documented in --help that the builder currently ignores). The chaos gate proves the pattern works — extend it.

9Generate the drift-prone lists instead of hand-copying them

Add a tiny build script (e.g., node scripts/export-builder-manifest.js) that dumps models, default tool names, prompt-template frontmatter, agents, and profiles to a JSON block embedded in the HTML — and a doc check that fails when the embedded snapshot is stale. Today three separate hand-maintained copies (PROMPT_REGISTRY, DEFAULT_TOOL_NAMES, the model/agent dropdowns) each carry a "keep in sync" comment; comments are promises, checks are mechanisms.

10Add "explain this command" + save/restore of configurations

Two learning aids: (a) an expandable per-flag breakdown under the terminal — each emitted flag as a row with a one-line meaning and a risk badge (read / write / shell / recovery), so the command becomes a vocabulary lesson; and (b) persist the current form to localStorage (plus a "Reset to defaults" button and optional URL-hash sharing), so a page refresh doesn't destroy ten minutes of careful configuration — the most demoralizing possible beginner experience.

Suggested sequencing

  1. Now (bug fixes, low risk): items 1–3.
  2. Next (highest beginner payoff): items 5, 6, 8.
  3. Then (structure and durability): items 4, 7, 9, 10.

Everything above is scoped to docs/command-builder.html (plus one optional script for item 9). No runner or bridge behavior changes are required, so none of the transport/auth invariants are touched.
