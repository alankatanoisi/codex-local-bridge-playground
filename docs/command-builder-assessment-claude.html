Command Builder — Critical Assessment

Claude's independent review of docs/command-builder.html against the runner's actual runtime, judged specifically through a beginner-programmer lens.

Reviewed file: docs/command-builder.html (2,165 lines) Checked against: bin/local-bridge-runner.js, src/runner/tool-*.js Date: 2026-07-07

Verdict: The command builder is genuinely thoughtful and, in its presets, is one of the most beginner-friendly parts of this project. But it is not yet optimized for a true novice. Two problems dominate: (1) it hand-copies facts from the runtime and has already drifted out of sync — most importantly it never exposes the ask_user_question tool that the runner turns on by default, and silently drops it (plus the worktree tools) the moment you customize the tool list; and (2) it presents an expert-sized wall of ~40 advanced options with heavy jargon (“ledger”, “flight recorder”, “fork from session”, “chaos-ok”) and little plain-language teaching. There is also a real, shipping CSS bug that breaks the two-column layout. None of this makes it unusable — but a beginner is more likely to feel lost than guided.

1. How I reviewed it

I read the whole builder page, then read the runner's real entry point (bin/local-bridge-runner.js) and the tool system (tool-catalog.js, tool-profiles.js, tool-registry.js) so I could compare the command the builder generates against the flags and tools the runner actually accepts and exposes. In other words: does the builder tell you the truth about what the runner will do? I did not open the pre-existing assessment file — this is a fresh, independent read.

2. What already works well for a beginner

  • Presets are excellent. Nine one-click starting points, each with a plain-English name (“Read-only exploration”, “Edit with confirmation”) and a ready-written prompt. This is the single most novice-friendly feature and should be pushed even harder.
  • The permission “style” dropdown hides raw flags. A beginner picks “Look only” or “Edit — ask before each change” instead of reasoning about --plan vs --accept-edits vs --dont-ask. Each choice shows a one-line hint. Good abstraction.
  • Advanced options are collapsed by default. The scary stuff is behind a <details> panel, so the first screen stays calm.
  • The generated command tells you where to run it. It prefixes cd /path && …, which directly answers a beginner's #1 question: “which folder am I supposed to be in?”
  • Live safety guardrails. The “chaos combo” conflict actually disables the copy button until you resolve it, and shell-on shows a warning banner. Real protective behavior, not just text.
  • Honest about risk. It repeatedly reminds you that traces can contain secrets and that shell lets the model run terminal commands. That matches this repo's safety-first tone.

3. Critical findings

Ordered most-severe first. Findings 1–3 are factual mismatches with the runtime; finding 4 is a real rendering bug.

High The builder never exposes ask_user_question, and silently deletes it when you customize tools

The runtime's isToolVisible() (in src/runner/tool-profiles.js) exposes every tool by default except four: apply_patch, bash, manage_shell_jobs, and lsp_query. That means the real default tool set is 16 tools and includes ask_user_question (the model's way to stop and ask you a multiple-choice question) and the worktree tools enter_worktree / exit_worktree.

The builder's capability catalog lists only 13 tools and omits ask_user_question entirely:

// docs/command-builder.html — DEFAULT_TOOL_NAMES
list_files, read_file, search_text, glob, git_status,
manage_tasks, spawn_agent, list_worktrees, run_skill,
edit_file, write_file, undo, undo_edit
// missing vs runtime default: ask_user_question, enter_worktree, exit_worktree

Why this matters for a beginner

As long as you leave the tool checkboxes alone, no --tools flag is emitted and the runner quietly uses its full 16-tool default — so it works by luck. But the instant you tick or untick any tool box, the builder writes an explicit --tools a,b,c… list built only from its checkboxes. That list has no way to include ask_user_question, so you silently lose the model's ability to ask you clarifying questions — exactly the safety net a novice most wants — without any indication it happened.

Fix

Add ask_user_question, enter_worktree, and exit_worktree to the catalog and to DEFAULT_TOOL_NAMES so the builder's idea of “default” matches the runtime's.

High “Default” tool state misrepresents what the runner actually exposes

In the form, enter_worktree and exit_worktree render unchecked, implying they're off. At runtime they are on by default. So the checkbox panel — the one place a beginner looks to answer “what can this agent do to my files?” — is inaccurate in both directions: it shows some default-on tools as off, and hides one default-on tool (ask_user_question) completely. Trust in the tool panel is misplaced.

Fix

Drive the checkbox defaults from the same rule the runtime uses (“everything except apply_patch, bash, manage_shell_jobs, lsp_query”) rather than a separate hand-written list.

Medium Hand-mirrored data will keep drifting (models, templates, tools)

Three separate places in the builder duplicate runtime facts: the model dropdown, the PROMPT_REGISTRY object (a copy of each template's recommended permissions/tools), and DEFAULT_TOOL_NAMES. The tool list has already drifted (findings 1–2). The model list is the next to rot — it is a static <select> with no tie to what the bridge will accept, so a beginner can pick a model that silently 400s. A code comment in the file even admits the registry must be “kept in sync” by hand.

Fix

Either generate these three data structures from the runtime at build/commit time, or add a tiny test that fails when they diverge (see improvement #2). For a solo learner, the test is the cheaper win.

Medium Real CSS bug: the two-column layout selector is missing

Around line 68 the .layout { } selector line was dropped, leaving an orphaned CSS block with no selector: 

.header-links {
  …
}
                    ← the ".layout {" line is missing here
  display: grid;
  grid-template-columns: 1fr 1fr;
  gap: 1.5rem;
  align-items: start;
}

Because the rule has no selector, the browser discards the entire block. The form and the generated-command panel were designed to sit side by side; with the rule gone they stack in one column and the “sticky” command panel loses its purpose. It still functions, but it does not look as intended — and a beginner would have no idea why.

Fix

One line: restore .layout { before display: grid;.

Low Flags exist in the runner that the builder can't produce

The runner accepts a few flags the builder has no control for, e.g. --list-agents, --list-profiles, and the runner eval / runner worktrees list subcommands. These are discovery/utility commands rather than run options, so their absence is defensible — but a short “Handy commands” box would help a beginner discover them instead of reading --help.

4. Beginner-experience critique

Setting correctness aside, is this optimized for a novice? Partly. The top of the page is, the bottom is not.

Aspect Beginner impact
Advanced panel is one giant list of ~40 controls Once opened, there's no sub-grouping a novice can navigate by goal. “Session & context” alone mixes session health, startup context, task hygiene, ledger utilities, budgets, agents, and traces. It reads like a reference, not a guide.
Undefined jargon ledger, flight recorder trace level, fork from session id, chaos-ok, workspace fingerprint, compaction, ghostAfter appear with little or no plain-language definition. This repo's own rules say “define jargon once when you use it.”
No “what will this actually do?” summary The builder outputs a command but never says, in one sentence, “This will read your files and suggest edits, asking before each change.” A beginner has to mentally decode the flags to know the blast radius.
No pre-flight / “is the bridge running?” help The command assumes the bridge is already up on 127.0.0.1:11437. If it isn't, the beginner gets a raw connection error with no guidance from the page.
Accessibility & density Several labels/hints sit at 0.72rem in --muted grey on dark — low contrast and small. Status is often conveyed by color alone (red conflict, orange caution).

5. Ten suggested improvements

Prioritized. Pills mark rough effort: Quick (minutes), Medium (an afternoon), Bigger (a focused session).

1Fix the tool drift so the builder tells the truth Quick

Add ask_user_question, enter_worktree, and exit_worktree to the capability catalog and to DEFAULT_TOOL_NAMES. This closes findings 1 and 2 — the highest-value change on the list, because it stops the builder from silently removing the model's ability to ask you questions.

2Add a “builder matches runtime” test Medium

Write one node:test that imports the runtime's default-visible tool set and the model/template facts, then asserts the builder's mirrored constants match. When the runtime changes, the test fails loudly instead of the docs drifting silently. For a solo learner this is the durable fix behind improvement #1.

3Restore the .layout CSS selector Quick

Add the missing .layout { line so the form and command panel render side-by-side as designed. One line, immediate visual correctness.

4Add a plain-English “What this command will do” summary Medium

Above or below the generated command, render one sentence assembled from the current settings, e.g. “Reads files in this folder and proposes edits, pausing to ask you before each file change. Shell is off.” This translates flags into consequences — the thing a beginner actually needs.

5Add a one-time “Before you run this” checklist Quick

A small collapsible box near the copy button: “1) Is the local bridge running? 2) Are you in the playground folder? 3) New here — start with a Read-only preset.” Include the exact command to start the bridge. This removes the most common first-run failure (bridge not up) before it happens.

6Group Advanced options by goal, with short intros Medium

Split the mega-panel into a few nested <details> sections titled by intent — “Attach files”, “Limit cost & length”, “Resume earlier work”, “Debugging & logs”. Each gets a one-line “use this when…” intro so a beginner scans by question, not by flag.

7Add hover/inline glossary for jargon Medium

Give the scary terms (ledger, trace, fork, compaction, chaos-ok) a small “?” tooltip or a linked glossary at the bottom. Define each once, in one sentence, in beginner terms. This is a direct application of this repo's “define jargon once” rule.

8Add a “Beginner mode” toggle that hides everything non-essential Bigger

A top-level switch that collapses the page to just: prompt, presets, permission style, folders, and the command. Everything else stays one click away. Default it ON. The current design hides advanced options but still shows a dense middle; a real beginner mode would show maybe eight controls total.

9Make the tool panel state-aware and explain the auto-changes Medium

When enabling shell auto-checks bash, or when the model dropdown warns about Opus temperature, surface a tiny inline note explaining why a box just changed. Right now the form mutates itself silently, which is confusing when you're still learning cause and effect.

10Improve contrast and add non-color status cues Quick

Nudge the smallest hint text up from 0.72rem and lighten --muted for body copy, and pair color-coded warnings with an icon or word (“Blocked”, “Caution”) so meaning survives for color-blind users and low-quality screens. Low effort, broad benefit.
