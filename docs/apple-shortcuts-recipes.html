🍎 Apple Shortcuts Recipes for the Bridge Runner

Five ready-to-build macOS Shortcuts that turn bin/local-bridge-runner.js into one-tap tools: project tours, Siri Q&A, code review from the share sheet, dry-run change plans, and a scheduled morning repo digest.

What is happening under the hood?

Every recipe below uses the same trick as the example shortcut you built: a Run Shell Script action that changes into the playground folder and calls the runner. The runner talks to the local bridge (the VS Code extension listening on localhost:11437), which supplies your Claude Code credentials. Nothing here needs an API key — but it does need the bridge running.

Jargon check: a Shortcut is a small automation you build in the macOS Shortcuts app (in your Applications folder). A Quick Action is a shortcut that appears when you right-click a file or folder in Finder. The share sheet is the share menu (the square-with-arrow icon) in most Mac apps.

Read this first — five things that make or break Shortcuts

  1. The bridge must be running. Open VS Code with the Claude Local Bridge extension active before running any shortcut. If the bridge is down, the runner prints a connection error instead of an answer — annoying, but harmless.
  2. Shortcuts cannot answer yes/no questions. When you run the runner in Terminal, it pauses and asks before editing files or running shell commands. A shortcut has no keyboard attached to that prompt, so it would just hang. That is why every recipe below sticks to read-only profiles, plan mode (a dry run), or a tightly constrained git-only shell. None of these shortcuts can modify your files.
  3. Shortcuts does not know where node lives. The Run Shell Script action starts a minimal shell without your Terminal setup. Every script below begins with an export PATH=... line covering Homebrew on Apple Silicon (/opt/homebrew/bin) and Intel (/usr/local/bin). If node still is not found, run which node in Terminal and add that folder to the line.
  4. Keep the output clean. The runner prints tips and progress on stderr (the error channel) and the answer on stdout (the normal channel). Shortcuts can mix the two together, so the scripts use --log-level quiet and 2>/dev/null (which discards the error channel) so only the answer reaches the next action.
  5. Every recipe has a spending cap. --max-cost-usd stops the run if the estimated cost passes the limit, and --max-wall-clock-ms stops runs that go too long. Shortcuts fire fast and sometimes accidentally (hello, Siri), so caps are non-negotiable here.

1 📖 Explain Any Project

Finder Quick Actionread-onlygreat first build

An upgrade of your example shortcut. Instead of always explaining the playground, this one explains whichever folder you right-click in Finder. The folder path arrives as the shortcut input, and the script hands it to the runner with --cwd — the flag that tells the runner which project folder its tools may look inside.

The --profile review-only flag loads a built-in capability profile that allows reading, listing, and searching files while hard-denying edit, write, patch, and shell tools. Even if the model asked to edit something, the profile says no.

Build it in the Shortcuts app

📥 Receive Files and Folders from Quick Actions, Share Sheet
If there's no input: Ask For Files. In shortcut settings, enable “Use as Quick Action” → “Finder”.
›_ Run Shell Script
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "$1" \
  --profile review-only \
  --log-level quiet \
  --max-cost-usd 0.50 \
  "Give me a beginner-friendly tour of this project: what it is, \
what the main folders do, and where a newcomer should start reading." \
  2>/dev/null
Shell: zsh
Input: Shortcut Input
Pass Input: as arguments  (the folder path becomes $1)
⏹ Stop and output Shell Script Result
If there's nowhere to output: Respond Shell Script Result
Why this is safe The review-only profile denies every write and shell tool at the permission layer, and the runner’s path confinement keeps reads inside the --cwd folder. Sensitive files like .env, keys, and .ssh are blocked by the deny matrix on top of that.
One gotcha Right-click a real project folder, not your home folder or Downloads — the runner validates --cwd at startup and refuses system-level directories, and pointing it at a huge non-project folder just wastes tokens.

2 🙋 Ask My Repo (Siri-friendly)

Siri / Spotlightread-onlyQ&A

A question box for your own codebase. Run it from the menu bar, Spotlight, or by saying “Hey Siri, Ask My Repo.” An Ask for Text action collects your question, and the script passes it to the runner as $1. Because the question is quoted as "$1", punctuation and apostrophes in your question cannot break the command.

Good questions to try: “Where is the permission logic for the bash tool?”, “What does --task-scope actually change?”, “Which tests cover safety.js?”

Build it in the Shortcuts app

💬 Ask for Text with prompt “What do you want to know about the repo?”
›_ Run Shell Script
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --profile review-only \
  --log-level quiet \
  --max-cost-usd 0.50 \
  "$1" \
  2>/dev/null
Shell: zsh
Input: Provided Input  (the answer from Ask for Text)
Pass Input: as arguments
⏹ Stop and output Shell Script Result
If there's nowhere to output: Respond Shell Script Result — this is what lets Siri read the answer back.
Variation worth trying Add --prompt-template explore before "$1" to prepend the built-in “explore” template, which nudges the model to search the codebase before answering instead of guessing from the question alone.

3 🔍 Review Shared Code

Share sheetstdin pastingprompt template

Select code anywhere — an editor, a web page, Mail — share it (or copy it), and get a code review back. This recipe shows off a runner feature the other recipes don’t: when text is piped to the runner’s stdin (the standard input stream a program can read from), the runner automatically attaches it to your prompt as pasted context. In Shortcuts, setting Pass Input: to stdin does exactly that piping for you.

It also uses --prompt-template review, the built-in “findings-first” review template stored in src/runner/prompts/review.md, which frames the model to lead with concrete findings on correctness, safety, maintainability, and missing tests.

Build it in the Shortcuts app

📥 Receive Text from Share Sheet, Quick Actions
If there's no input: Get Clipboard — so it also works on whatever you last copied.
›_ Run Shell Script
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --profile review-only \
  --prompt-template review \
  --log-level quiet \
  --max-cost-usd 0.50 \
  "Review the code pasted below. Explain each finding in plain language \
a beginner can follow, and say which findings matter most." \
  2>/dev/null
Shell: zsh
Input: Shortcut Input
Pass Input: to stdin  (the shared code is piped in and becomes pasted context)
⏹ Stop and output Shell Script Result
Tip: insert a Quick Look action before this one to read long reviews in a scrollable window.
Bonus: reviewing a file already in the repo Swap the stdin approach for --include-file src/runner/safety.js to attach a bounded, secret-scrubbed copy of a repo file instead. --include-file refuses files outside the project, blocked paths, and anything over 50 KB.

4 🗺️ Dry-Run Change Plan

Ask-for-textplan modeclipboard

Describe a change you’re considering — “add a --version flag to the runner CLI” — and get back a concrete step-by-step plan without anything being executed. That is what --plan does: the runner inspects the code, and any action the model proposes is answered with a dry-run result instead of being performed. It is the safest way to preview what an agent session would do.

The plan lands on your clipboard, so you can paste it straight into a real Terminal session (or a Cursor/Claude chat) when you’re ready to actually make the change.

Build it in the Shortcuts app

💬 Ask for Text with prompt “What change are you considering?”
›_ Run Shell Script
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --plan \
  --log-level quiet \
  --max-cost-usd 0.75 \
  --max-wall-clock-ms 180000 \
  "Plan this change without making it: $1. List the files you would touch, \
the steps in order, the checks to run afterwards, and any risks." \
  2>/dev/null
Shell: zsh
Input: Provided Input
Pass Input: as arguments
📋 Copy to Clipboard Shell Script Result
⏹ Stop and output Shell Script Result
If there's nowhere to output: Show Result Shell Script Result
Plan mode is a preview, not a promise A plan is the model’s best guess before doing the work. When you later run the change for real (in Terminal, where you can approve each edit), expect small differences — that’s normal.

5 🌅 Morning Repo Digest

Scheduled automationgit-only shellJSON outputnotification

The most advanced recipe: a Shortcuts Personal Automation (Automation tab → “Time of Day”) that runs every morning, has the runner execute read-only git commands to see what changed recently, and delivers a short digest as a notification.

This is the only recipe that enables the shell tool — and it shows how the runner’s layered safety model makes that acceptable for an unattended run:

  • --allow-shell turns the bash tool on (it is hidden by default).
  • --profile git-readonly-shell then pins bash to read-only git commands (git status/log/diff/show/branch/rev-parse/describe) via a command allowlist, and denies every edit, write, and patch tool.
  • --dont-ask skips confirmation prompts — required because nobody is at the keyboard at 9 AM. It only applies to tools already enabled; it cannot enable anything new.
  • Cost and wall-clock caps bound the blast radius if a run goes sideways.

It also demonstrates --output-format json: the runner emits a machine-readable result object, and Shortcuts extracts the finalText field with a Get Dictionary Value action — much more reliable than scraping terminal text.

Build it in the Shortcuts app

⏰ Automation: Time of Day → 9:00 AM, Daily → Run Immediately
Build the actions below as a normal shortcut first, then point the automation at it.
›_ Run Shell Script
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --allow-shell \
  --dont-ask \
  --profile git-readonly-shell \
  --output-format json \
  --log-level quiet \
  --max-cost-usd 0.50 \
  --max-wall-clock-ms 180000 \
  "Use read-only git commands (git log, git diff --stat) to see what \
changed in this repo in the last 2 days. Give me a 5-sentence digest: \
what changed, anything surprising, and what I might look at today." \
  2>/dev/null
Shell: zsh
Input: (none)  ·  Pass Input: as arguments
🔤 Get Dictionary from Shell Script Result
🔎 Get Dictionary Value finalText
🔔 Show Notification Dictionary Value with title “Repo digest”
Alternative endings: Append to a Note, or Send Message to yourself.
Handle these flags with care --allow-shell --dont-ask is only reasonable here because the git-readonly-shell profile constrains what the shell can do. Never copy this flag pair into a shortcut without a restrictive --profile, and never add --accept-edits to it — the runner treats all three together as a “chaos combo” and refuses unless you explicitly acknowledge the risk.
Prerequisite reminder Scheduled automations run whether or not you remembered to open VS Code. If the bridge is down, the JSON parse step fails and the notification never fires — a quiet failure. If you want a loud one, add an “If / Otherwise” action around the dictionary step.

Why none of these shortcuts edit files

You could build a shortcut with --accept-edits that lets the model change files without confirmation. This guide deliberately doesn’t include one. A shortcut runs without a visible terminal, so you would not see what was edited until after the fact — the exact opposite of the “read the proposed change, then approve it” habit that keeps agent runs safe. Do editing sessions in Terminal (or via the command builder at docs/command-builder.html), where confirmations work and you can watch the diff.

Also remember the standing policy note from the threat model: the bridge reuses your Claude Code OAuth credentials for personal research. Keep these shortcuts personal, low-volume, and on your own machine.
